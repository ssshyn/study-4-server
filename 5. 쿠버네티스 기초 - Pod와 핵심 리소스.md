## 1. 쿠버네티스가 필요한 이유

```
Docker Compose의 한계
	- 단일 서버에서만 동작 (여러 서버로 분산 불가)
	- 컨테이너 죽으면 수동으로 재시작
	- 트래픽에 따라 자동 스케일 아웃 불가
	- 롤링 업데이트, 롤백 어려움
	  
쿠버네티스(k8s)가 해결:
	- 여러 서버(Node)에 컨테이너 분산 배포
	- 컨테이너 장애 시 자동 재시작
	- 트래픽 기반 자동 스케일 (HPA)
	- 롤링 업데이트 / 롤백 내장
	- 선언적 배포 (원하는 상태를 선언 → k8s가 맞춤)
```

## 3. 핵심 리소스
### 3-1. Pod - 가장 작은 배포 단위

```yaml
# pod.yaml
apiVersion: v1
kind: Pod
metadata:
	name: myapp-pod
	labels:
		app: myapp
		version: "1.0"
spec:
	containers:
		- name: app
		  image: myapp:1.0.0
		  ports:
			  - containerPort: 8080
		  resources:
			requests:
				memory: "256Mi"   # 최소 보장 자원
				cpu: "250m"       # 100m = 1 CPU
			limits:
				memory: "512Mi"   # 최대 허용 지원
				cpu: "500m"
		  env:
			- name: SPRING_PROFILES_ACTIVE
			  value: "prod"
			- name: DB_PASSWORD
			  valueFrom:
				  secretKeyRef:
					  name: db-secret
					  key: password
		  livenessProbe:      # 컨테이너가 살아있는지 확인
			  httpGet:
				  path: /actuator/health/liveness
				  port: 8080
		  readinessProbe:     # 트래픽 받을 준비가 됐는지 확인
			  httpGet:
				  path: /actuator/health/readiness
				  port: 8080
			  initialDelaySeconds: 10
			  periodSeconds: 5
```

```
Pod는 직접 배포하지 않는다.
Pod가 죽으면 재시작 안 됨 → Deployment를 통해 배포
```

### 3-2. Deployment - Pod 관리와 롤링 업데이트

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
	name: myapp
	namespace: production
spec:
	replicas: 3                            # Pod 개수
	selector:
		matchLabels:
			app: myapp                     # 이 Deployment가 관리할 Pod 선택
	strategy:
		type: RollingUpdate
		rollingUpdate:
			maxSurge: 1                    # 업데이트 중 최대 추가 Pod 수
			maxUnavailable: 0              # 업데이트 중 최소 유지할 Pod 수
	template:                              # Pod 템플릿 (위의 Pod spec과 동일)
		metadata:
			labels:
				app: myapp
		spec:
      containers:
        - name: app
          image: myapp:1.0.0
          ports:
            - containerPort: 8080
          resources:
            requests:
              memory: "256Mi"
              cpu: "250m"
            limits:
              memory: "512Mi"
              cpu: "500m"
          livenessProbe:
            httpGet:
              path: /actuator/health/liveness
              port: 8080
            initialDelaySeconds: 30
          readinessProbe:
            httpGet:
              path: /actuator/health/readiness
              port: 8080
            initialDelaySeconds: 10
```

```bash
# 배포
kubectl apply -f deployment.yaml

# 업데이트 (이미지 변경)
kubectl set image deployment/myapp app=myapp:1.1.0

# 롤아웃 상태 확인
kubectl rollout status deployment/myapp

# 롤백
kubectl rollout undo deployment/myapp
kubectl rollout undo deployment/myapp --to-revision=2

# 이력 확인
kubectl rollout history deployment/myapp
```

### 3-3. Service - Pod에 안정적인 네트워크 접근

```
Pod는 재시작되면 IP가 바뀜 → Service가 안정적인 진입점 제공
Service는 라벨 셀렉터로 Pod를 선택, 로드 밸런싱

Service 종류:
	ClusterIP (기본): 클러스터 내부에서만 접근
	NodePort: 각 노드의 특정 포트로 외부 접근 (개발/테스트용)
	LoadBalancer: 클라우드의 LB를 생성 (AWS ELB 등)
```

```yaml
# service.yaml
apiVersion: v1
kind: Service
metadata:
	name: myapp-service
spec:
	selector:
		app: myapp
	type: ClusterIP
	ports:
		- protocol: TCP
		  port: 80             # Service 포트
		  targetPort: 8080     # Pod 포트
---
# 외부 트래픽용 (AWS ELB 자동 생성)
apiVersion: v1
kind: Service
metadata:
	name: myapp-lb
	annotations:
		service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
spec:
	selector:
		app: myapp
	type: LoadBalancer
	ports:
		- port: 80
		  targetPort: 8080
```

### 3-4. Ingress - HTTP 라우팅

```yaml
# ingress.yaml - 하나의 LB로 여러 서비스 라우팅
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
	name: myapp-ingress
	annotations:
		nginx.ingress.kubernetes.io/rewrite-target: /
		cert-manager.io/cluster-issuer: letsencrypt-port   # SSL 자동화
spec:
	ingressClassName: nginx
	tls:
		- hosts:
			  - api.myapp.com
		  secretName: myapp-tls
	rules:
		- host: api.myapp.com
		  http:
			  paths:
				  - path: /api/orders
				    patchType: Prefix
				    backend:
					    service:
						    name: order-service
						    port:
							    number: 80
				  - path: /api/users
				    patchType: Prefix
				    backend:
					    service:
						    name: user-service
						    port:
							    number: 80
```

## 4. ConfigMap과 Secret

### 4-1. ConfigMap - 일반 설정

```yaml
# configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
	name: myapp-config
data:
	SPRING_PROFILES_ACTIVE: "prod"
	LOG_LEVEL: "INFO"
	# 파일 형태도 가능
	application.yml: |
		server:
			port: 8080
		logging:
			level:
				root: INFO
```

```yaml
# Deployment에서 사용
spec:
	containers:
		- envFrom:
			  - configMapRef:
					name: myapp-config       # 전체 키를 환경변수로
		  volumeMounts:
			  - name: config-volume
			    mountPath: /app/config
	volumes:
		- name: config-volume
		  configMap:
			  name: myapp-config              # 파일로 마운트
```

### 4-2. Secret - 민감 정보

```bash
# Secret 생성 (base64 인코딩 자동)
kubectl create secret generic db-secret \
	--from-literal=username=appuser \
	--from-literal=password=supersecret
	
# 또는 파일로
kubectl create secret generic tls-secret \
	--from-file=tls.crt=cert.pem
	--from-file=tls.key=key.pem
```

```yaml
# Secret YAML (값은 base64 dlszheld)
apiVersion: v1
kind: Secret
metadata:
	name: db-secret
type: Opaque
data:
	username: YXBwdXNlcg==     # echo -n "appuser" | base64
	password: c3VwZXJzZWNyZXQ= # echo -n "supersecret" | base64
```

> 주의: Secret은 etcd에 base64로 저장됨 (암호화 아님)
> 운영에서는 AWS Secrets Manager + External Secrets Operator 또는 Sealed Secrets (Bitnami) 사용 권장.

## 5. HPA - 수평 자동 스케일링

```yaml
# hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
	name: myapp-hpa
spec:
	scaleTargetRef:
		apiVersion: apps/v1
		kind: Deployment
		name: myapp
	minReplicas: 2          # 최소 Pod 수
	maxReplicas: 10         # 최대 Pod 수
	metrics:
		- type: Resource
		  resource:
			  name: cpu
			  target:
				  type: Utilization
				  averageUtilization: 70    # CPU 70% 초과 시 스케일 아웃
		- type: Respource
		  resource:
			  name: memory
			  target:
				  type: AverageValue
				  averageValue: 400Mi
```

## 6. 자주 쓰는 kubectl 명령어

```bash
# 리소스 조회
kubectl get pods                          # Pod 목록
kubectl get pods -n production            # 네임스페이스 지정
kubectl get pods -o wide                  # 노드 정보 포함
kubectl get all                           # 모든 리소스
kubectl get events --sort-by=.lastTimestamp  # 이벤트 최신순

# 상세 정보
kubectl describe pod myapp-pod-xxxxx      # Pod 상세 (이벤트 포함)
kubectl describe deployment myapp

# 로그
kubectl logs myapp-pod-xxxxx
kubectl logs myapp-pod-xxxxx -c app       # 특정 컨테이너 (멀티 컨테이너 Pod)
kubectl logs -f deployment/myapp          # Deployment 전체 로그 스트리밍
kubectl logs --previous myapp-pod-xxxxx  # 이전 컨테이너 로그 (재시작 후)

# Pod 내부 접속
kubectl exec -it myapp-pod-xxxxx -- sh

# 포트 포워딩 (로컬 테스트)
kubectl port-forward pod/myapp-pod-xxxxx 8080:8080
kubectl port-forward service/myapp-service 8080:80

# 리소스 삭제
kubectl delete pod myapp-pod-xxxxx
kubectl delete -f deployment.yaml

# 네임스페이스
kubectl create namespace production
kubectl config set-context --current --namespace=production  # 기본 ns 변경
```

## 7. Namespace - 논리적 환경 분리

```yaml
# 팀/환경별로 네임스페이스 분리
production  # 운영
staging     # 스테이징
development # 개발
monitoring  # Prometheus, Grafana
```

```bash
# 네임스페이스별 리소스 할당 (ResourceQuota)
kubectl apply -f - <<EOF
apiVersion: v1
kind: ResourceQuota
metadata:
	name: production-quota
	namesapce: production
spec:
	hard:
		requests.cpu: "10"
		requests.memory: 20Gi
		limits.cpu: "20"
		limits.memory: 40Gi
		pods: "50"
EOF
```
