## 1. 왜 ELK?

```
분산 시스템에서의 로그 문제:

서비스 5개 x 인스턴스 3개 = 15개 서버에 로그가 흩어짐
	→ 에러 추적하려면 15개 서버에 SSH로 들어가서 grep?
	→ 특정 시간대 전체 서비스 로그를 한눈에 볼 수 없음
	→ 로그가 시간 순서대로 정렬이 안됨
	
ELK:
	모든 로그를 한 곳에 모아서 → 전문 검색 + 시각화 
	"5분 전에 어떤 서비스에서 ERROR가 몇 번 났는지"를 대시보드에서 즉시 확인
```

---
## 2. ELK/EFK 스택 구성

```
ELK Stack:
	E - Elasticsearch: 로그 저장 & 검색 엔진
	L - Logstash:      로그 수집 & 변환 파이프라인
	K - Kibana:        시각화 & 대시보드
	
EFK Stack (경량버전):
	E - Elasticsearch
	F - Fluent Bit / Fluentd: Logstash 대비 경량 수집기
	K - Kibana
	
흐름:
	앱 → Fluent Bit (수집/파싱) → Elasticsearch (저장) → Kibana (조회)
```

## 3. Elasticsearch 기초

### 3-1. 핵심 개념

```
Index:      RDB의 테이블. 로그 인덱스는 보통 날짜별로 분리
			myapp-logs-2024.01.15 (날짜 롤오버)

Document:   RDB의 Row. Json 형태로 저장

Shard:      인덱스를 나누는 단위 (병렬 처리)
			Primary Shard + Replica Shard

Cluster → Node → Index → Shard → Document
```

### 3-2. 인덱스 매핑 (Schema 정의)

```json
PUT /myapp-logs-000001
{
  "mappings": {
    "properties": {
      "@timestamp":  { "type": "date" },
      "level":       { "type": "keyword" },
      "service":     { "type": "keyword" },
      "traceId":     { "type": "keyword" },
      "spanId":      { "type": "keyword" },
      "message":     { "type": "text", "analyzer": "standard" },
      "exception":   { "type": "text" },
      "userId":      { "type": "long" },
      "duration":    { "type": "long" },
      "httpMethod":  { "type": "keyword" },
      "httpPath":    { "type": "keyword" },
      "httpStatus":  { "type": "integer" }
    }
  },
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1,
    "index.lifecycle.name": "myapp-logs-policy"  // ILM 정책
  }
}
```

## 4. Spring Boot 로그 설정

### 4-1. Logback JSON 포맷

```xml
<!-- logback-spring.xml -->
<configuration>
  <springProfile name="prod">
    <!-- JSON 형식으로 출력 (Elasticsearch가 파싱하기 쉬움) -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
      <encoder class="net.logstash.logback.encoder.LogstashEncoder">
        <includeMdcKeyName>traceId</includeMdcKeyName>
        <includeMdcKeyName>spanId</includeMdcKeyName>
        <includeMdcKeyName>userId</includeMdcKeyName>
        <customFields>{"service":"order-service","environment":"prod"}</customFields>
      </encoder>
    </appender>
    
    <root level="INFO">
      <appender-ref ref="CONSOLE" />
    </root>
  </springProfile>

  <springProfile name="local">
    <!-- 로컬에서는 사람이 읽기 쉬운 형식 -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
      <encoder>
        <pattern>%d{HH:mm:ss.SSS} [%thread] %-5level [%X{traceId}] %logger{36} - %msg%n</pattern>
      </encoder>
    </appender>
    <root level="DEBUG">
      <appender-ref ref="CONSOLE" />
    </root>
  </springProfile>
</configuration>
```

### 4-2. MDC로 컨텍스트 정보 추가
```java
// 요청마다 TraceId, UserId를 MDC에 저장 → 모든 로그에 자동 포함
@Component
public class LoggingFilter implements OncePerRequestFilter {
	@Override
	protected void doFilterInternal(HttpServletRequest request,
									HttpServletResponse response,
									FilterChain chain) throws IOException, ServletException {
		try {
			// TraceId: 요청 전체를 추적하는 고유 ID
			String traceId = Optional.ofNullable(request.getHeader("X-Trace-Id")).orElse(UUID.randomUUID().toString().replace("-","").substring(0, 16));
			
			MDC.put("traceId", traceId);
			MDC.put("httpMethod", request.getMethod());
			MDC.put("httpPath", request.getRequestURI());
			
			response.setHeader("X-Trace-Id", traceId); // 응답에도 포함
			
			chain.doFilter(request, response);
		} finally {
			MDC.clear();  // 스레드 풀 재사용 시 이전 값 오염 방지
		}
	}
}

// 서비스에서 userId 추가
@Service
public class OrderService {
	public Order createOrder(Long userId, ...) {
		MDC.put("userId", userId.toString());
		log.info("주문 생성 시작");
		// 이후 모든 로그에 userId가 자동 포함됨
	}
}
```

JSON 로그 출력 예시:

## 5. Fluent Bit 설정 (Kubernetes)

```yaml
# fluent-bit-configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: fluent-bit-config
  namespace: logging
data:
  fluent-bit.conf: |
    [SERVICE]
        Flush         5
        Log_Level     info
        Parsers_File  parsers.conf

    # 입력: Kubernetes 컨테이너 로그 수집
    [INPUT]
        Name              tail
        Path              /var/log/containers/*.log
        multiline.parser  docker, cri
        Tag               kube.*
        Refresh_Interval  5
        Mem_Buf_Limit     50MB
        Skip_Long_Lines   On

    # 필터: Kubernetes 메타데이터 추가 (파드명, 네임스페이스 등)
    [FILTER]
        Name                kubernetes
        Match               kube.*
        Kube_URL            https://kubernetes.default.svc:443
        Merge_Log           On        # JSON 로그를 파싱해서 필드로 분리
        Keep_Log            Off
        K8S-Logging.Parser  On
        K8S-Logging.Exclude On

    # 필터: monitoring 네임스페이스 제외
    [FILTER]
        Name    grep
        Match   kube.*
        Exclude $kubernetes['namespace_name'] monitoring

    # 출력: Elasticsearch로 전송
    [OUTPUT]
        Name            es
        Match           kube.*
        Host            elasticsearch.logging.svc.cluster.local
        Port            9200
        Index           k8s-logs
        Logstash_Format On          # 날짜별 인덱스 (k8s-logs-2024.01.15)
        Logstash_Prefix k8s-logs
        Time_Key        @timestamp
        Retry_Limit     5
        Replace_Dots    On
```

## 6. Index Lifecycle Management (ILM)

```json
// 로그 인덱스 자동 관리 정책
// Hot → Warm → Cold → Delete

PUT _ilm/policy/myapp-logs-policy
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_size": "50gb",     // 50GB 넘으면 새 인덱스
            "max_age": "1d"         // 1일 지나면 새 인덱스
          }
        }
      },
      "warm": {
        "min_age": "7d",            // 7일 후 warm으로
        "actions": {
          "shrink": { "number_of_shards": 1 },  // 샤드 축소
          "forcemerge": { "max_num_segments": 1 }
        }
      },
      "cold": {
        "min_age": "30d",           // 30일 후 cold로
        "actions": {
          "freeze": {}              // 읽기 전용, 압축
        }
      },
      "delete": {
        "min_age": "90d",           // 90일 후 삭제
        "actions": {
          "delete": {}
        }
      }
    }
  }
}
```

## 7. Kibana 대시보드
### 7-1. 주요 기능

```
Discover:
	원본 로그 조회 및 필터링
	KQL(Kibana Query Language)로 검색
	예: level: ERROR and service: "order-service"
	
Visualize:
	집계 쿼리를 차트로 시각화
	- 시간별 에러 건수 (Line Chart)
	- 서비스별 로그 비율 (Pie Chart)
	- 평균 응답 시간 (Gauge)
	  
Dashboard:
	여러 Visualization을 하나의 화면에 실시간 자동 갱신 (30초, 1분 ...)
	
Alerts:
	조건 충족 시 Slack/이메일 알림
	예: 5분 내 ERROR 10건 이상 → 슬랙 알림
```

### 7-2. KQL 검색 예시

```kql
# 에러 로그
level: ERROR

# 특정 서비스의 에러
level: ERROR AND service: "order-service"

# 응답 시간 1초 이상
duration >= 1000

# 특정 사용자 관련 로그
userId: 12345 AND @timestamp >= now-1h

# 예외 메시지 포함
exception: *NullPointerException*

# 특정 API 에러
httpPath: /api/orders AND httpStatus >= 500
```

## 8. OpenSearch (AWS 환경)

```
AWS에서는 Elasticsearch 대신 OpenSearch Service 사용 권장
	(Elasticsearch 라이선스 이슈로 AWS가 포크한 오픈 소스)
	
OpenSearch = Elasticsearch와 거의 동일한 API → Kibana → OpenSearch Dashboards
 
AWS OpenSearch Service 구성:
	- 관리형 (설치/패치/백업 자동) 
	- VPC 내부 배포 가능
	- CloudWatch 연동
	- 세밀한 IAM 권한 제어
	  
비용 최적화:
	OR1 인스턴스 (UltraWarm): 핫 데이터 대비 87% 절감
	Cold Storage: S3에 저장, 필요 시 재인덱싱
```
