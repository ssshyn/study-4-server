
## 1. 로그 vs 메트릭 vs 트레이스 - 관찰 가능성 3요소

```
Logs (로그):         "무슨 일이 일어났는가" - 이벤트 기록
						→ ELK, CloudWatch Logs

Metrics (메트릭):    "얼마나, 얼마나 자주" - 수치 데이터
						→ Prometheus + Grafana, CloudWatch Metrics

Traces (추적):       "어디서 얼마나 걸렸는가" - 분산 요청 흐름
						→ Jaeger, Zipkin, AWS X-Ray

세 가지를 함께 사용해야 문제를 완전히 파악할 수 있음

예시:
	알람: "주문 API 응답 시간 급증"
	
	Metrics → 어떤 시간에, 얼마나 늘었는지
	Traces  → 어떤 서비스에서 병목인지
	Logs    → 병목 서비스에서 구체적으로 무슨 에러가 났는지
```

## 2. Prometheus
### 2-1. 개념

```
Pull 방식: Prometheus가 주기적으로 앱의 /metrics 엔드포인트를 수집
			(Push 방식인 CloudWatch와 반대)
			
Time Series DB: 시계열 데이터 저장에 최적화
				(metric_name{label="value"} timestamp value)
				
PromQL: 메트릭 조회/집계를 위한 쿼리 언어
```

### 2-2. Spring Boot Actuator + Micrometer 연동

```yaml
# build.gradle
implementation 'org.springframework.boot:spring-boot-starter-actuator'
implementation 'io.micrometer:micrometer-registry-prometheus'
```

```yaml
# application.yml
management:
	endpoints:
		web:
			exposure:
				include: health, info, prometheus, metrics
	endpoint:
		prometheus:
			enabled: true
		health:
			show-details: when-authorized
	metrics:
		tags:
			application: ${spring.application.name} # 모든 메트릭에 앱 이름 태그
			environment: ${spring.profiles.active}
		distribution:
			percentiles-histogram:
				http.server.requests: true     # 응답 시간 히스토그램
			slo:
				http.server.requests: 100ms, 500ms, 1s  # SLO 임계값
```

`/actuator/prometheus` 접속 시 출력 예시:

### 2-3. 커스텀 메트릭

```java
@Service
public class OrderService {
    private final MeterRegistry meterRegistry;
    
    // Counter: 누적 건수 (주문 생성 수, 에러 수)
    private final Counter orderCreatedCounter;
    private final Counter orderFailedCounter;
    
    // Timer: 실행 시간
    private final Timer paymentProcessingTimer;
    
    // Gauge: 현재 상태 (대기 중인 주문 수)
    private final AtomicInteger pendingOrderCount = new AtomicInteger(0);
    
    public OrderService(MeterRegistry meterRegistry) {
        this.meterRegistry = meterRegistry;
        
        this.orderCreatedCounter = Counter.builder("order.created.total")
            .description("총 주문 생성 건수")
            .tag("service", "order")
            .register(meterRegistry);
        
        this.orderFailedCounter = Counter.builder("order.failed.total")
            .description("주문 실패 건수")
            .register(meterRegistry);
        
        this.paymentProcessingTimer = Timer.builder("payment.processing.duration")
            .description("결제 처리 시간")
            .publishPercentiles(0.5, 0.9, 0.99)
            .register(meterRegistry);
        
        // Gauge는 참조만 등록 (값은 람다로 동적 조회)
        Gauge.builder("order.pending.count", pendingOrderCount, AtomicInteger::get)
            .description("대기 중인 주문 수")
            .register(meterRegistry);
    }
    
    public Order createOrder(CreateOrderCommand command) {
        try {
            pendingOrderCount.incrementAndGet();
            
            Order order = paymentProcessingTimer.recordCallable(() -> {
                return processOrderWithPayment(command);
            });
            
            orderCreatedCounter.increment();
            return order;
            
        } catch (Exception e) {
            orderFailedCounter.increment(
                Tags.of("reason", e.getClass().getSimpleName())
            );
            throw e;
        } finally {
            pendingOrderCount.decrementAndGet();
        }
    }
}
```

```yaml
global:

rule_files:
  - "alert-rules.yml"

        - targets: ['alertmanager:9093']

      - role: pod

        target_label: pod
      - source_labels: [__meta_kubernetes_namespace]

      - targets: ['mysql-exporter:9104']
```

## 4. PromQL — 쿼리 언어

```promql
# 분당 HTTP 요청 수 (rate: 초당 평균)
rate(http_server_requests_seconds_count[1m]) * 60

# 5xx 에러율
sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m]))
  /
sum(rate(http_server_requests_seconds_count[5m]))

# 서비스별 P99 응답 시간
histogram_quantile(0.99,
  sum(rate(http_server_requests_seconds_bucket[5m]))
  by (le, application, uri)
)

# JVM 힙 사용률
jvm_memory_used_bytes{area="heap"}
  /
jvm_memory_max_bytes{area="heap"}

# CPU 사용률
rate(process_cpu_usage[1m]) * 100

# DB 커넥션 풀 사용률
hikaricp_connections_active
  /
hikaricp_connections_max

# 지난 1시간 평균 응답 시간
avg_over_time(
  rate(http_server_requests_seconds_sum[5m])
    /
  rate(http_server_requests_seconds_count[5m])
[1h:5m])
```

## 5. 알람 규칙

```yaml
# alert-rules.yml
groups:
  - name: application
    rules:
      # 5xx 에러율 1% 초과
      - alert: HighErrorRate
        expr: |
          sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m]))
            /
          sum(rate(http_server_requests_seconds_count[5m]))
          > 0.01
        for: 2m                      # 2분 지속 시 알람
        labels:
          severity: critical
        annotations:
          summary: "에러율 높음: {{ $labels.application }}"
          description: "에러율 {{ $value | humanizePercentage }}"

      # P99 응답 시간 1초 초과
      - alert: HighLatency
        expr: |
          histogram_quantile(0.99,
            sum(rate(http_server_requests_seconds_bucket[5m])) by (le, application)
          ) > 1
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "P99 응답 시간 높음: {{ $labels.application }}"

      # JVM 힙 90% 초과
      - alert: JvmMemoryHigh
        expr: |
          jvm_memory_used_bytes{area="heap"}
            /
          jvm_memory_max_bytes{area="heap"}
          > 0.9
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "JVM 힙 메모리 부족: {{ $labels.application }}"

      # Pod 재시작
      - alert: PodRestarting
        expr: |
          increase(kube_pod_container_status_restarts_total[1h]) > 3
        for: 1m
        labels:
          severity: warning
        annotations:
          summary: "Pod 재시작 빈번: {{ $labels.pod }}"
```

```yaml

global:
  slack_api_url: 'https://hooks.slack.com/services/...'

route:
  group_by: [alertname, application]
  group_wait: 30s
  group_interval: 5m

  - name: slack-warning
    slack_configs:
      - channel: '#alerts-warning'
        title: '⚠️ {{ .GroupLabels.alertname }}'
        send_resolved: true
```

## 8. SLI / SLO / SLA

```
SLI (Service Level Indicator): 서비스 수준을 측정하는 지표
	- 가용성: 성공 응답 수 / 전체 요청 수
	- 응답 시간: P99 < 1초
	- 에러율: 5xx 비율

SLO (Service Level Objective): SLI의 목표값
	- 가용성 99.9% (월 43분 다운타임 허용)
	- P99 응답 시간 < 1초
	- 에러율 < 0.1%
	  
SLA (Service Level Agreement): SLO를 고객과 계약
	"99.9% 가용성 미달 시 환불 등 패널티"
	
Error Budget (에러 예산):
	SLO 99.9% → 0.1% = 월 43분의 다운타임이 "예산"
	에러 예산이 소진되면 새 기능 배포 중단 → 안정화 집중
	
Grafana에서 SLO 시각화:
	이번 달 가용성: 99.95%
	남은 에러 예산: 67% (33% 소진)
	이 추세면 월말까지: 예산 초과 가능성 15%
```
