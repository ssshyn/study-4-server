
## 1. CI/CD

```
CI (Continuous Integration): 코드 변경 시 자동으로 빌드 + 테스트
CD (Continuous Delievery): CI 이후 스테이징까지 자동 배포
CD (Continuous Deployment): 프로덕션까지 자동 배포

목표:
	- "코드를 push하면 알아서 배포된다."
	- 빠른 피드백 루프 (코드 버그를 push 후 5분 내에 알 수 있음)
	- 수동 배포 실수 제거
	- 작고 자주 배포 → 위험 감소
	  
일반적인 파이프라인:
	코드 Push → 빌드 → 단위 테스트 → 통합 테스트 → 이미지 빌드 & Push → 스테이징 배포 → E2E 테스트 → 운영 배포
```

## 2. GitHub Actions

### 2-1. 기본 개념

```yaml
# .github/workflows/ci.yml

on:                                    # 트리거: 언제 실행할까
	push:
		branches: [main, develop]
	pull_request:
		branches: [main]
		
jobs:                                  # 병렬 실행 가능한 작업 단위
	test:
		runs-on: ubuntu-latest         # 실행 환경
		steps:                         # 순차 실행 단계
			- uses: actions/checkout@v4  # 코드 체크아웃
			- name: Run tests
			  run: ./gradlew test
```

### 2-2. 실전 - Spring Boot CI 파이프라인

```yaml
# .github/workflows/ci-cd.yml
name: CI/CD Pipeline

on:
	push:
		branches: [main]
	pull_request:
		branches: [main]
	
env:
	AWS_REGION: ap-northeast-2
	ECR_REPOSITORY: myapp
	ECS_SERVICE: myapp-service
	ECS_CLUSTER: myapp-cluster
	
jobs:
	# -----------------------------------------
	# Job 1: 테스트
	# -----------------------------------------
	test:
		name: Build & Test
		runs-on: ubuntu-latest
		
		services:
			mysql:
				image: mysql:8.0
				env:
					MYSQL_ROOT_PASSWORD: test
					MYSQL_DATABASE: testdb
				options: >-
					--health-cmd="mysqladmin ping"
					--health-interval=10s
					--health-timeout=5s
					--health-retries=5
				ports:
					- 3306:3306
			redis:
				image: redis:7.2-alpine
				ports:
					- 6379:6379
					  
		steps:
			- uses: actions/checkout@v4
			
			- name: Set up JDK 17
			  uses: actions/setup-java@v4
			  with:
				  java-version: '17'
				  distribution: 'temurin'
				  
			- name: Cache Gradle packages
			  uses: actions/cache@v4
			  with:
				  path: ~/.gradle/caches
				  key: ${{ runner.os }}-gradle-${{ hashFiles('**/*.gradle*') }}
				  restore-keys: ${{ runner.os }}-gradle-
				  
			- name: Run tests
			  env:
				  SPRING_PROFILES_ACTIVE: test
				  DB_URL: jdbc:mysql://localhost:3306/testdb
				  DB_USERNAME: root
				  DB_PASSWORD: test
				  REDIS_HOST: localhost
				run: ./gradlew test --no-daemon
				
			- name: Publish test report
			  uses: EnricoMi/publish-unit-test-result-action@v2
			  if: always()
			  with:
				  files: build/test-results/**/*.xml
			
			- name: Upload coverage to Codecov
			  uses: codecov/codecov-action@v3
			
	# -----------------------------------------
	# Job 2: 이미지 빌드 & ECR Push (main 브랜치만)
	# -----------------------------------------
	build-and-push:
		name: Build Docker Image
		needs: test                    # test 성공 후에만 실행
		if: github.ref == 'refs/heads/main'
		runs-on: ubuntu-latest
		
		outputs:
			image-tag: ${{ steps.meta.outputs.version }}
		
		steps:
			- uses: actions/checkout@v4
			
			- name: Set up JDK 17
			  uses: actions/setup-java@v4
			  with:
          java-version: '17'
          distribution: 'temurin'
      
      - name: Build jar
        run: ./gradlew bootJar --no-daemon
      
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ${{ env.AWS_REGION }}
      
      - name: Login to Amazon ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v2
      
      - name: Docker meta (태그 생성)
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ steps.login-ecr.outputs.registry }}/${{ env.ECR_REPOSITORY }}
          tags: |
            type=sha,prefix=,suffix=,format=short
            type=raw,value=latest
      
      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
          
    # -----------------------------------------
	# Job 3: ECS 배포
	# -----------------------------------------
	deploy:
		name: Deploy to ECS
		needs: build-and-push
		runs-on: ubuntu-latest
		environment: production    # Github Enviroments 승인 게이트
		
		steps:
			- name: Configure AWS credentials
			  uses: aws-actions/configure-aws-credentials@v4
			  with:
				  aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
				  aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
				  aws-region: ${{ env.AWS_REGION }}
				  
			- name: Download task definition
			  run: |
				  aws ecs describe-task-definition \
				   --task-definition myapp \
				   --query taskDefinition \
				   > task-definition.json
			- name: Update ECS task definition with new image
			  id: task-def
			  uses: aws-actions/amazon-ecs-render-task-definition@v1
			  with:
				  task-definition: task-definition.json
				  container-name: app
				  image: ${{ needs.build-and-push.outputs.image-tag }} 
				  
			- name: Deploy Amazon ECS service
			  uses: aws-actions/amazon-ecs-deploy-task-definition@v1
			  with:
				  task-definition: ${{ steps.task-def.outputs.task-definition }}
				  service: ${{ env.ECS_SERVICE }}
				  cluster: ${{ env.ECS_CLUSTER }}
				  wait-for-service-stability: true # 배포 완료까지 대기
```

### 2-3. PR 체크 워크플로우

```yaml
# .github/workflows/pr-check.yml
name: PR Check

on:
  pull_request:
    branches: [main, develop]

jobs:
  code-quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
      
      # 정적 코드 분석
      - name: Run Checkstyle
        run: ./gradlew checkstyleMain checkstyleTest
      
      # 단위 테스트 + 커버리지
      - name: Run tests with coverage
        run: ./gradlew test jacocoTestReport
      
      # 커버리지 70% 미만이면 실패
      - name: Check coverage threshold
        run: ./gradlew jacocoTestCoverageVerification
      
      # PR에 테스트 결과 코멘트
      - name: Comment PR with test results
        uses: EnricoMi/publish-unit-test-result-action@v2
        if: always()
        with:
          files: build/test-results/**/*.xml
          comment_mode: always
```

### 3-1. GitHub Flow (단순, 스타트업/소규모)

```
main ─────────────────────────────────────────→
         │           │
    feature/A    feature/B
    (PR → main)  (PR → main)
    즉시 배포      즉시 배포

특징:
  - main만 배포 브랜치
  - feature 브랜치 → PR → main → 즉시 배포
  - 단순하지만 여러 버전 동시 지원 불가
```

### 3-2. Git Flow (대규모, 릴리즈 주기 명확한 경우)

```
main (태그: v1.0, v1.1)
  │
develop ──────────────────────────────────→
  │           │            │
feature/A  feature/B   release/1.1
                            │
                        hotfix/urgent
                            │ (main, develop에 머지)
```

### 3-3. 환경별 파이프라인

```
develop 브랜치 push → 자동 스테이징 배포
main 브랜치 push     → 자동 운영 배포 (or 승인 후)
태그 push (v1.2.0)   → 버전 태깅 + 릴리즈 노트 생성
```

## 4. Secrets 관리

```
GitHub Secrets 계층:
	Repository Secrets: 해당 레포만 사용
	Environment Secrets: 특정 환경(production)에만 적용
	Organization Secrets: 조직 전체 레포에서 사용
	
사용 패턴:
	AWS_ACCESS_KEY_ID       → Repository Secret (CI/CD 권한)
	AWS_SECRET_ACCESS_KEY   → Repository Secret
	SLACK_WEBHOOK_URL       → Organization Secret (알림 공통)

	prod DB 비밀번호는 GitHub Secrets에 넣지 않음
	→ AWS Secrets Manager에 저장하고
		ECS Task Role로 접근 (CI/CD 파이프라인 불필요)
```

## 5. 배포 전략

### 5-1. Rolling Update (기본)

```
3개 Pod 순차 교체:

Before: [v1] [v1] [v1]
Step 1: [v2] [v1] [v1]  ← v1 하나를 v2로 교체
Step 2: [v2] [v2] [v1]
Step 3: [v2] [v2] [v2]
After: [v2] [v2] [v2]

장점: 추가 인프라 불필요, 무중단
단점: 교체중 v1/v2 혼재, 문제 발생 시 일부 사용자만 영향
```

### 5-2. Blue/Green (안전환 전환)

```
Blue (현재 운영): [v1] [v1] [v1]  ← 트래픽 100%
Green (새 버전): [v2] [v2] [v2]  ← 준비 완료

전환: LB가 Green으로 트래픽 전환
문제 시: LB를 Blue로 즉시 롤백

장점: 즉각 롤백, 전환 시간 최소
단점: 인프라 2배 비용, 전환 중 DB 스키마 호환성 주의
```

### 5-3. Canary (점진적 롤아웃)

```
단계별 트래픽 전환:
	v1: 100%, v2: 0%
	v1: 90%, v2: 10%   ← 소수 사용자로 검증
	v1: 50%, v2: 50%
	v1: 0%, v2: 100%
	
에러율, 응답 시간 모니터링하며 진행
문제 발생 시 v1으로 즉시 복귀

AWS에서: CodeDeploy + ALB Weighted Target Groups
k8s에서: Argo Rollouts, Flagger
```

## 6. 슬랙 알림 연동

```yaml
# 배포 성공/실패 알림
- name: Notify Slack on success
  if: success()
  uses: slackapi/slack-github-action@v1
  with:
	payload: |
		  {
			  "text": "*배포 성공*: ${{ github.repository }}",
			  "attachments": [{
				  "color": "good",
				  "fields": [
					  {"title": "브랜치", "value": "${{ github.ref_name }}", "short": true},
					  {"title": "커밋", "value": "${{ github.sha }}", "short": true},
					  {"title": "배포자", "value": "${{ github.actor }}", "short": true}
				  ]
			  }]
		  }
	env:
		SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
		
- name: Notify Slack on failure
  if: failure()
  uses: slackapi/slack-github-action@v1
  with:
	  payload: |
		  {
			  "text": "*배포 실패*: ${{ github.repository }}",
			  "attachments": [{
			   "color": "danger",
			   "text" "Actions URL: ${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}"
			  }]
		  }
	  env:
		  SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
```
