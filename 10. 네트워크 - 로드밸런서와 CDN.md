## 1. 요청이 서버에 도달하는 과정

```
사용자가 https://api.myapp.com/orders를 요청할 때:

1. DNS 조회 (Route 53)
	api.myapp.com → ALB IP 주소 반환
	
2. TLS Handshake
   브라우저 <> ALB 간 SSL 협상 (인증서 검증)
   
3. ALB (Application Load Balancer)
   요청을 뒤쪽 서버 (ECS/EC2)로 분배
   
4. 서버 처리
   Spring Boot 앱이 요청 처리

5. 응답 반환
   서버 → ALB → 사용자
```

## 2. 로드밸런서 종류
### 2-1. AWS 로드밸런서 비교

```
CLB (Classic LB) → 구버전, 사용 지양

ALB (Application LB) - L7 (HTTP/HTTPS):
	- URL 경로 기반 라우팅
	  /api/orders → Order Service
	  /api/users  → User Service
	- 호스트 기반 라우팅
	  api.myapp.com   → API 서버
	  admin.myapp.com → 관리자 서버
	- 요청/응답 헤더 조작 가능
	- WebSocket 지원
	- gRPC 지원 (HTTP/2)
	- 권장: 대부분의 웹 서비스
	  
NLB (Network LB) - L4 (TCP/UDP):
	- IP:Port 기반 라우팅 (패킷 레벨)
	- 초저지연 (ALB보다 낮은 레이턴시)
	- 정적 IP 지원
	- 초당 수백만 요청 처리
	- 권장: gRPC 서비스 간 통신, 게임 서버, IoT
```

### 2-2. ALB 리스너 규칙

```
HTTPS:443 리스너 규칙 (우선순위 순서):

우선순위 1: Host = api.myapp.com AND Path = /api/admin/* 
	→ Admin Target Group (인증 필요 → Cognito 연동)
	
우선순위 2: Host = api.myapp.com AND Path = /api/orders*
	→ Order Service Target Group
	
우선순위 3: Host = api.myapp.com AND Path = /api/users*
	→ User Service Target Group
	
우선순위 4: 기본 (위 규칙에 해당 없는 경우)
	→ 404 고정 응답
```

### 2-3. Target Group - 서버 풀

```
Target Group:
	- 요청을 받을 서버/컨테이너 묶음
	- Health Check로 비정상 서버 자동 제외
	  
Health Check 설정:
	Protocol: HTTP
	Path: /actuator/health
	Port: 8080
	Healthy threshold: 2    # 2번 연속 성공 → 정상
	Unhealthy threshold: 3  # 3번 연속 실패 → 비정상, 트래픽 제외
	Interval: 30s
	Timeout: 5s
	
Deregistration Delay (Draining):
	Target을 제외할 때 즉시 제거하지 않고
	기존 연결이 완료될 때까지 300초 대기
	→ 배포 중 사용자 요청이 끊기지 않음
```

## 3. CDN - CloudFront
### 3-1. CDN이 필요한 이유

```
사용자: 부산
서버: 서울 리전

사용자가 10MB 이미지 요청:
	부산 → 서울 → 처리 → 서울 → 부산
	지연 시간: 수십 ms + 10MB 전송 시간

CDN 사용 시:
	부산 인근 엣지 서버에 캐시됨
	부산 → 부산 엣지 → 즉시 응답
	지연 시간: 수 ms
	서버 부하 없음

CloudFront 엣지 서버: 전 세계 600개 잇아
```

### 3-2. CloudFront 배포 설정

```
오리진 (Origin):
	S3 버킷 → 정적 파일 (이미지, JS, CSS, HTML)
	ALB    → 동적 API
	
캐싱 동작 (Cache Behavior):
	기본: *.js, *.css, 이미지 → 7일 캐싱
	/api/* → 캐시 없음 (TTL=0, 오리진으로 직접 전달)
```

```
CloudFront URL 예시:
	정적 파일: https://d1234abcd.cloudfront.net/images/banner.jpg
	커스텀 도메인: https://cdn.myapp.com/images/banner.jpg
	
캐시 무효화:
	배포 후 JS/CSS 캐시 제거
	aws cloudfront create-invalidation \
		--distribution-id ABCDE12345
		--paths "/js/*" "/css/*"
		
	비용 최적화: 파일명에 해시 포함으로 자동 캐시 버스팅
	main.a1b2c3d4.js → 내용 바뀌면 해시가 바뀌어 자동 갱신 
```

```
🚨 캐시 버스팅 (Cache Busting)

브라우저나 CDN이 오래된(캐시된) 파일을 계속 쓰지 않고, 새 버전의 파일을 강제로 다시 만드는 기법
```

## 4. DNS - Route 53
### 4-1. 주요 레코드 타입

```
A 레코드:  도메인 → IPv4 주소
			api.myapp.com → 54.12.456.789

CNAME:    도메인 → 다른 도메인
			www.myapp.com → myapp.com
			(A 레코드 대상이 바뀌어도 CNAME은 수정 불필요)
			
ALIAS:    Route 53 전용. AWS 리소스로 직접 매핑
			(ALB IP가 바뀌어도 자동 반영)
			
TXT:      도메인 소유 확인, SPF 레코드

MX:       이메일 서버 (수신)
```

### 4-2. 라우팅 정책

```
Simple:     단순 IP 반환

Weighted:         가중치 기반 분산 (Canary 배포)
	myapp-v1.com  → 가중치 90
	myapp-v2.com  → 가중치 10
	
Failover:    Primary 장애 시 Secondary로
	Primary: 서울 리전 ALB
	Failover: Tokyo 리전 ALB

Latency:     응답 시간 기준 가까운 리전으로
	사용자가 일본 → Tokyo 리전
	사용자가 한국 → Seoul 리전
	
Geolocation:  사용자 위치 기반
	한국 사용자 → 한국 서버
	미국 사용자 → 미국 서버
	기타 → 기본 서버
	
Health Check 연동:
	Route 53이 직접 서버 Health Check
	비정상 서버는 DNS에서 제거
```

## 5. SSL/TLS - HTTPS
### 5-1. TLS Handshake 과
```
클라이언트                          서버 (ALB)
    │                                  │
    │── Client Hello ─────────────────→│  (지원하는 TLS 버전, 암호화 방식)
    │                                  │
    │←─ Server Hello ──────────────────│  (선택된 암호화 방식 + 인증서)
    │←─ Certificate ───────────────────│
    │                                  │
    │  [인증서 검증: CA 체인 확인]       │
    │                                  │
    │── Client Key Exchange ──────────→│  (대칭키 암호화에 쓸 세션키 교환)
    │── Change Cipher Spec ──────────→│
    │── Finished ────────────────────→│
    │                                  │
    │←─ Change Cipher Spec ────────────│
    │←─ Finished ──────────────────────│
    │                                  │
    │══════ 암호화된 통신 시작 ═════════│
```
### 5-2. AWS Certificate Manager (ACM)

```bash
# 도메인 인증서 발급 (무료, 자동 갱신)
aws acm request-certificate \
  --domain-name "*.myapp.com" \
  --validation-method DNS \
  --region ap-northeast-2

# Route 53에 CNAME 레코드 추가해서 도메인 소유권 검증
# → 검증 완료 후 ALB에 인증서 연결
```

### 5-3. SSL Termination (SSL 종료)

```
패턴 1: ALB에서 SSL 종료 (가장 일반적)
	클라이언트 -HTTPS-> ALB -HTTP-> 서버
	서버는 암호화 부담 없음, 인증서 관리가 ALB에서 중앙화
	
패턴 2: End-to-End 암호화
	클라이언트 -HTTPS-> ALB -HTTPS-> 서버
	서버까지 암호화. 규정 준수(PCI-DSS) 요구 시
	
패턴 3: SSL Passthrough (NLB 사용)
	클라이언트 -HTTPS-> NLB -HTTPS-> 서버 (패킷 그대로 전달)
	NLB가 내용을 볼 수 없음. 서버가 인증서 관리
```

## 6. API Gateway 패턴

```
클라이언트 → API Gateway → 마이크로서비스들

API Gateway 역할:
	1. 인증/인가: JWT 토큰 검증 (서비스마다 중복 구현 불필요)
	2. Rate Limiting: IP/사용자별 요청 제한
	3. 라우팅: 경로 기반으로 서비스 라우팅
	4. 응답 캐싱: 반복 요청에 캐시된 응답 반환
	5. 로깅/모니터링: 모든 요청/응답 중앙 기록
	6. 서킷 브레이커: 서비스 장애 격리
	7. 프로토콜 변환: REST → gRPC 변환
	   
AWS API Gateway vs Spring Cloud Gateway:
	AWS API Gateway: 서버리스, 관리 불필요, 비용은 요청당
	Spring Cloud GW: 직접 운영, 코드로 유연한 제어, ECS/EKS 배
```
### Spring Cloud Gateway 설정

```yaml
# application.yml
spring:
  cloud:
    gateway:
      routes:
        - id: order-service
          uri: lb://ORDER-SERVICE      # 서비스 디스커버리 사용
          predicates:
            - Path=/api/orders/**
          filters:
            - name: CircuitBreaker
              args:
                name: order-cb
                fallbackUri: forward:/fallback/orders
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 100
                redis-rate-limiter.burstCapacity: 200
            - AddRequestHeader=X-Gateway-Request, true
            
        - id: user-service
          uri: lb://USER-SERVICE
          predicates:
            - Path=/api/users/**
          filters:
            - name: JwtAuthentication  # 커스텀 JWT 검증 필터
```

## 7. 서비스 메시 - Istio (선택 학습)

```
마이크로서비스가 많아질 때 공통 인프라 문제를 해결:

Sidecar 패턴:
	각 Pod에 Envoy Proxy를 사이드카로 붙임
	→ 모든 서비스 간 통신이 Envoy를 통과
	
Istio가 제공하는 것:
	- 상호 TLS (mTLS): 서비스 간 자동 암호화
	- 트래픽 관리: 카나리 배포, 서킷 브레이커
	- 관찰 가능성: 분산 추적, 메트릭 자동 수집
	- 정책 적용: 서비스별 Rate Limit
	  
단점:
	- 복잡성 높음
	- 사이드카로 인한 리소스 오버헤드
	- 학습 곡선 가파름
	  
	→ 서비스 수가 20개 미만이면 API Gateway로 충분
```

```
🚨 카나리 배포 (Canary Deployment)

새 버전을 전체 사용자에게 한 번에 배포하지 않고, 일부 사용자(소수)에게만 먼저 배포하여 문제가 없는지 확인한 뒤, 점점 비율을 늘려가며 전체로 확대하는 배포 전략 
```
