## 1. 분산 추적 필요성

```
모놀리식에서의 요청 추적:
	Controller → Service → Repository → DB
	단일 스택 트레이스, 단일 로그로 추적 가능
	
MSA에서의 요청 추적:
	Client → API GW → Order → Payment → Inventory → Notification
	5개 서비스, 각각 다른 로그 → 어디서 얼마나 걸렸는지 파악 불가
	
문제 시나리오:
	주문 API 응답 시간: 3초
	Order Service: 50ms
	Payment Service: 2800ms  ← 여기가 병목인데 어떻게 아나?
	Inventory Service: 100ms
	
→ 분산 추적: 단일 요청이 여러 서비스를 거치는 전체 경로를 시각화
```
---
## 2. 핵심 개념

```
Trace: 단일 요청의 전체 흐름 (루트에서 리프까지)
	TraceId: 전체 요청을 식별하는 고유 ID (모든 서비스가 공유)

Span: Trace 내의 단일 작업 단위
	SpanId: 이 Span의 ID
	ParentSpanId: 부모 Span의 ID (루트는 없음)
	시작 시각, 종료 시각, 태그, 이벤트
	
시각화:
  [TraceId: abc123]
  ├── [Span: API GW]         0ms ~ 3100ms
  │   ├── [Span: Order Svc]  10ms ~ 3090ms
  │   │   ├── [Span: DB Query] 20ms ~ 70ms
  │   │   └── [Span: Payment Svc] 80ms ~ 2880ms  ← 병목!
  │   │       ├── [Span: 카드사 API] 90ms ~ 2850ms
  │   │       └── [Span: DB Save]  2860ms ~ 2880ms
  │   └── [Span: Inventory] 2890ms ~ 2990ms
  └── [Span: Notification]  3000ms ~ 3090ms (비동기)
```

## 3. OpenTelemetry (OTel)
### 3-1. 왜 OpenTelemetry인가

```
예전: Jaeger SDK, Zipkin SDK, Datadog SDK 각각 따로 구현
	→ 벤더 종속, 교체 시 코드 수정 필요
	
OpenTelemetry: CNCF가 주도하는 관찰 가능성 표준
	- 단일 API/SDK로 Logs, Metrics, Traces 수집
	- 벤더 중립: Jaeger, Zipkin, Datadog, Grafana Tempo 등으로 전송 가능
	- 코드 변경 없이 백엔드 교체
	  
흐름:
	앱 (OTel SDK) → OTel Collector → Jaeger / Grafana Tempo / Datadog
```

```
🚨벤더 종속(Vendor Lock-in)

특정 업체(벤더)의 기술/서비스/플랫폼에 너무 깊이 의존해서, 나중에 다른 업체로 바꾸고 싶어도 비용, 기술적 문제 때문에 사실상 못 바꾸게 되는 상태
```

### 3-2. Spring Boot + OpenTelemetry 설정

```gradle
// build.gradle
implementation 'io.opentelemetry.instrumentation:opentelemetry-spring-boot-starter'
// 또는 Micrometer Tracing (Spring Boot 3.x)
implementation 'io.micrometer:micrometer-tracing-bridge-otel'
implementation 'io.opentelemetry.instrumentation:opentelemetry-logback-mdc-1.0'
```

```yaml
# application.yml
management:
  tracing:
    sampling:
      probability: 1.0    # 100% 샘플링 (운영에서는 0.1 ~ 0.01)

spring:
  application:
    name: order-service

# OTel Collector로 전송
otel:
  exporter:
    otlp:
      endpoint: http://otel-collector:4318
  resource:
    attributes:
      service.name: order-service
      deployment.environment: prod
```

### 3-3. 자동 계측 (Auto-Instrumentation)

```
Spring Bood 자동 계측 대상:
	- HTTP 요청/응답 (Spring MVC, WebFlux)
	- DB 쿼리 (JDBC, JPA)
	- Redis 호출 (Lettuce, Jedis)
	- Kafka 프로듀서/컨슈머
	- 외부 HTTP 호출 (RestTemplate, WebClient, Feign)
	- 스케줄러 (@Scheduled)
	  
코드 변경 없이 위 항목 모두 Span으로 자동 기록됨
```

### 3-4. 수동 Span — 비즈니스 로직 추적

```java
@Service
public class OrderService {
    private final Tracer tracer;
    
    @Autowired
    public OrderService(Tracer tracer) {
        this.tracer = tracer;
    }
    
    public Order createOrder(CreateOrderCommand command) {
        // 커스텀 Span 시작
        Span orderSpan = tracer.nextSpan()
            .name("create-order")
            .tag("user.id", command.getUserId().toString())
            .tag("item.count", String.valueOf(command.getItems().size()))
            .start();
        
        try (Tracer.SpanInScope scope = tracer.withSpan(orderSpan)) {
            
            // 쿠폰 검증 Span
            Span couponSpan = tracer.nextSpan().name("validate-coupon").start();
            try (Tracer.SpanInScope cs = tracer.withSpan(couponSpan)) {
                validateCoupon(command.getCouponCode());
            } finally {
                couponSpan.end();
            }
            
            Order order = processOrder(command);
            
            orderSpan.tag("order.id", order.getId().toString());
            orderSpan.event("order-created");  // 타임스탬프 이벤트
            
            return order;
            
        } catch (Exception e) {
            orderSpan.tag("error", "true")
                     .tag("error.message", e.getMessage());
            throw e;
        } finally {
            orderSpan.end();
        }
    }
}
```

## 4. TraceId를 로그에 포함시키기

```java
// Logback MDC에 TraceId/SpanId 자동 추가
// micrometer-tracing이 자동으로 처리

// logback-spring.xml
<encoder class="net.logstash.logback.encoder.LogstashEncoder">
  <includeMdcKeyName>traceId</includeMdcKeyName>
  <includeMdcKeyName>spanId</includeMdcKeyName>
</encoder>
```

로그 + 트레이스 연결:
```json
{
  "@timestamp": "2024-01-15T14:23:01.234Z",
  "level": "ERROR",
  "traceId": "a1b2c3d4e5f6g7h8",   ← Jaeger에서 같은 ID로 검색
  "spanId": "x1y2z3",
  "service": "order-service",
  "message": "결제 서비스 호출 실패"
}
```

Kibana에서 ERROR 로그 발견 → traceId 복사 → Jaeger에서 전체 흐름 확인

## 5. OTel Collector 설정

```yaml
# otel-collector-config.yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

processors:
  batch:
    timeout: 1s
    send_batch_size: 1024
  
  # 샘플링 (운영 트래픽 조절)
  tail_sampling:
    decision_wait: 10s
    policies:
      - name: errors-policy
        type: status_code
        status_code: {status_codes: [ERROR]}  # 에러는 100% 수집
      - name: slow-traces-policy
        type: latency
        latency: {threshold_ms: 1000}         # 1초 이상 요청 수집
      - name: probabilistic-policy
        type: probabilistic
        probabilistic: {sampling_percentage: 10}  # 나머지는 10%만

exporters:
  # Jaeger로 전송
  jaeger:
    endpoint: jaeger:14250
    tls:
      insecure: true
  
  # Prometheus로 메트릭 전송
  prometheus:
    endpoint: "0.0.0.0:8889"
  
  # Elasticsearch로 로그 전송
  elasticsearch:
    endpoints: ["http://elasticsearch:9200"]

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch, tail_sampling]
      exporters: [jaeger]
    metrics:
      receivers: [otlp]
      processors: [batch]
      exporters: [prometheus]
```


## 7. AWS X-Ray (AWS 환경)

```
AWS 환경에서의 분산 추적:
  → X-Ray: AWS 네이티브, ECS/Lambda/ALB 자동 계측

Spring Boot + X-Ray:
implementation 'com.amazonaws:aws-xray-recorder-sdk-spring'

@Configuration
@EnableXRay
public class XRayConfig {
    @Bean
    public Filter xrayFilter() {
        return new AWSXRayServletFilter("myapp");
    }
}

# ALB → ECS 간 X-Ray 자동 추적
# Lambda → DynamoDB / S3 등 자동 추적
```

## 8. 실무 세팅 요약

```
소규모 (서비스 5개 이하):
  Spring Boot Actuator + Micrometer → Grafana Tempo (무료)
  Grafana에서 Tempo + Loki (로그) + Prometheus (메트릭) 통합 뷰

AWS 환경:
  OTel SDK → CloudWatch (통합, 추가 설치 최소)
  또는 OTel → Grafana Cloud (유료, 강력한 시각화)

대규모:
  OTel Collector → Jaeger (추적) + Prometheus (메트릭) + ELK (로그)
  3가지를 Grafana에서 통합 조회 (Correlation)
```
