## 2. JWT (JSON Web Token)
### 2-1. 구조

```
JWT = Header.Payload.Signature

Header: { "alg": "HS256", "typ": "JWT" } →
Base64URL
Payload: {
	"sub": "12345",                       // 사용자 ID
	"email": "hong@example.com",
	"roles": ["USER", "ADMIN"],
	"iat": 1710000000,                    // 발행 시각
	"exp": 1710003600                     // 만료 시각 (1시간 후)
}
Signature: HMAC-SHA256(header + "." + payload, secretKey)
```

### 2-2. 주의사항

```
Base64URL은 인코딩이지 암호화가 아님
	→ Payload는 누구나 디코딩 가능
	→ 민감 정보(비밀번호, 카드번호) Payload에 넣으면 안 됨
	
서명(Signature)은 변조 감지용:
	서버만 알고 있는 secretKey로 서명
	토큰이 변조되면 서명 검증 실패
```

### 2-3. Spring Boot JWT 구현

```java
// build.gradle
implementation 'io.jsonwebtoken:jjwt-api:0.12.3'
runtimeOnly 'io.jsonwebtoken:jjwt-impl:0.12.3'
runtimeOnly 'io.jsonwebtoken:jjwt-jackson:0.12.3'

@Component
public class JwtProvider {
	@Value("${jwt.secret}")
	private String secret;
	
	// 1시간
	private static final long ACCESS_TOKEN_EXPIRE_MS = 60 * 60 * 1000L;
	// 7일
	private static final long REFRESH_TOKEN_EXPIRE_MS = 7 * 24 * 60 * 60 * 1000L;
	
	private SecretKey getSigningKey() {
		return Keys.hmacShaKeyFor(Decoders.BASE64.decode(secret));
	}
	
	// 액세스 토큰 생성
	public String generateAccessToken(Long userId, List<String> roles) {
		return Jwts.builder()
				.subject(userId.toString())
				.claim("roles", roles)
				.issuedAt(new Date())
				.expiration(new Date(System.currentTimeMillis() + ACCESS_TOKEN_EXPIRE_MS))
				.signWith(getSigningKey())
				.compact();
	}
	
	// 토큰 검증 & 파싱
	public Claims parseToken(String token) {
		return Jwts.parser()
			.verifyWith(getSigningKey())
			.build()
			.parseSignedClaims(token)
			.getPayload();
	}
	
	public Long extractUserId(String token) {
		return Long.parseLong(parseToken(token).getSubject());
	}
}

// Spring Security 필터
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {
	@Override
	protected void doFilterInternal(HttpServletRequest request,
	Http ServletResponse response, FilterChain chain) throws IOException, ServletException {
		String token = extractToken(request);
		
		if (token != null) {
			try {
				Claims claims = jwtProvider.parseToken(token);
				Long userId = Long.parseLong(claims.getSubject());
				List<String> roles = claims.get("roles", List.class);
				
				// SecurityContext에 인증 정보 설정
				UsernamePasswordAuthenticationToken auth = new UsernamePasswordAuthenticationToken(userId, null, roles.stream().map(simpleGrantedAuthority::new).toList());
				
				SecurityContextHolder.getContext().setAuthentication(auth);
			} catch (ExpiredJwtException e) {
				response.setError(HttpStatus.UNAUTHORIZED.value(), "토큰 만료");
				return;
			} catch (JwtException e) {
				response.setError(HttpStatus.UNAUTHORIZED.value(), "유효하지 않은 토큰");
				return;
			}
		}
		
		chain.doFilter(request, response);
	}
	
	private String extractToken(HttpServletRequest request) {
		String header = request.getHeader("Authorization");
		if (header != null && header.startsWith("Bearer ")) {
			return header.substring(7);
		}
		return null;
	}
}
```

### 2-4. Refresh Token 전략

```
문제: 액세스 토큰 만료 시간을 너무 짧게 하면 UX 나쁨 너무 길게 하면 탈취 시 오래 사용됨

해결: 짧은 액세스 토큰 + 긴 리프레시 토큰

액세스 토큰: 1시간, JWT (Stateless)
리프레시 토큰: 7일, Redis에 저장 (Stateful, 강제 만료 가능)

흐름:
	로그인 → 액세스 + 리프레시 토큰 발급
	API 요청 → 액세스 토큰으로 인증
	액세스 토큰 만료 → 리프레시 토큰으로 새 액세스 토큰 발급
	로그아웃 → Redis에서 리프레시 토큰 삭제
	탈취 의심 → Redis에서 해당 리프레시 토큰 강제 삭제
	
리프레시 토큰 Redis 저장:
	Key: refresh_token:{userId}
	Value: 토큰 값
	TTL: 7일
```

## 3. OAuth 2.0 / OIDC
### 3-1. 왜 OAuth 인가

```
"카카오로 로그인" 같은 소셜 로그인 흐름

우리 서비스가 직접 비밀번호를 관맇자 ㅣ않음
	→ 카카오, 구글, 네이버가 인증 담당
	→ 우리는 인증 완료된 사용자 정보만 받음
	
보안 이점:
	- 비밀번호 저장 안 함 → 유출 위험 없음
	- 큰 서비스의 강력한 보안 인프라 활용
	- 사용자는 이미 알고 있는 계정으로 로그
```

### 3-2. Authorization Code Flow

```
사용자          우리 앱           카카오
  │                │                │
  │─ 로그인 클릭 ──→│                │
  │                │── 카카오 로그인 페이지로 리다이렉트 ──→│
  │◄─── 카카오 로그인 페이지 ────────────────────────────│
  │                │                │
  │─ 카카오 ID/PW 입력 ─────────────→│
  │                │                │
  │                │◄── Authorization Code 발급 ─────────│
  │                │                │
  │                │── Code로 Access Token 요청 ─────────→│
  │                │◄── Access Token 반환 ───────────────│
  │                │                │
  │                │── Access Token으로 사용자 정보 요청 ──→│
  │                │◄── 사용자 정보 반환 ────────────────│
  │                │                │
  │                │ (우리 서비스 회원가입/로그인 처리)
  │◄── JWT 발급 ────│
```

```java
@Service
public class OAuth2UserService extends DefaultOAuth2UserService {
	@Override
	public OAuth2User loadUser(OAuth2UserRequest request) {
		OAuth2User oAuth2User = super.loadUser(request);
		
		String provider = request.getClientRegistration().getRegistrationId(); // "kakao", "google"
		OAuth2UserInfo userInfo = OAuth2UserInfoFactory.of(provider, oAuth2User.getAttributes());
		
		// 소셜 계정으로 회원가입 or 로그인
		User user = userRepository.findByProviderAndProviderId(provider, userInfo.getId()).orElseGet(() -> signUp(provider, userInfo));
		
		return new CustomOAuth2User(user, oAuth2User.getAttributes());
	}
}
```

```java
// Spring Security - 보안 헤더 설정
@Configuration
public class SecurityConfig {
	@Bean
	public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
		http
			.headers(headers -> headers
			// XSS 방어
			.xssProtection(xss -> xss.headerValue(XXssProtectionHeaderWriter.HeaderValue.ENABLED_MODE_BLOCK)))
			// 클릭재킹 방어
			.frameOptions(frame -> frame.deny())
			// MIME 스니핑 방어
			.contentTypeOptions(Customizer.withDefaults())
			// HTTPS 강제 (1년)
			.httpStrictTransportSecurity(hsts -> hsts
				.maxAgeInSeconds(3153600)
				.includeSubDomains(true))
			// Content Security Policy
			.contentSecurityPolicy(csp -> csp.policyDirectives("default-src 'self'; script-src 'self' 'unsafe-inline'"))
			)
			// ...
			;
		return http.build();
	}
}
```

## 6. OWASP Top 10 - 자주 나오는 취약점

```
1. SQL Injection
	나쁜 예: "SELECT * FROM users WHERE id = " + userId ← 직접 문자열 연결
	좋은 예: PreparedStatement 또는 JPA (파라미터 바인딩)
	
2. XSS (Cross-Site Scripting)
	사용자 입력을 HTML에 그대로 출력 → 악성 스크립트 실행
	대응: 출력 시 HTML 이스케이프, CSP 헤더
	
3. Broken Authentication
	약한 비밀번호 정책, 세션 고정 공격
	대응: BCrypt 해싱, 로그인 후 세션 재생성
	
4. Insecure Direct Object Reference (IDOR)
   GET /orders/12345 → 다른 사람 주문 조회 가능?
   대응: 현재 로그인 사용자 소유 리소스인지 항상 검증
   
5. Security Misconfiguration
   H2 Console 운영에서 열려 있음
   Spring Actuator /env 외부 노출
   대응: 운영 환경 설정 정기 감사
   
6. Sensitive Data Exposure
   비밀번호를 평문으로 로그에 출력
   응답에 불필요한 민감 정보 포함
   대응: 로그 필터링, @JsonIgnore
   
7. CSRF
	다른 사이트에서 인증된 사용자 대신 요청 전송
	대응: CSRF 토큰, SameSite 쿠키
```

```java
// IDOR 방어 패턴
@GetMapping("/orders/{orderId}")
public OrderResponse getOrder(@PathVariable Long orderId, @AuthenticationPrincipal Long userId) {
	Order order = orderRepository.findById(orderId).orElseThrow(OrderNotFoundException::new);
	
	// 반드시 소유권 검증
	if (!order.getUserId().equals(userId)) {
		throw new AccessDeniedException("다른 사용자의 주문입니다.");
	}
	
	return OrderResponse.from(order);
}

// 더 나은 방법: Repository 단에서 userId로 조회
Order order = orderRepository.findByIdAndUserId(orderId, userId).orElseThrow(OrderNotFoundException::new);
```

## 7. 보안 체크리스트

```
코드: 
	□ SQL Injection: PreparedStatement / JPA 파라미터 바인딩 사용
	□ XSS: 출력 이스케이프, CSP 헤더
	□ IDOR: 모든 리소스 접근에 소유권 검증
	□ 민감 정보 로그 출력 없음
	□ 비밀번호 BCrypt 해싱
	
인증:
	□ JWT 서명 키 충분히 긴 랜덤 값 (256비트 이상)
	□ 리프레시 토큰 Redis 저장 + 강제 만료 가능
	□ 로그인 실패 횟수 제한 (Rate Limiting)
	
인프라:
	□ 모든 비밀값 Secrets Manager 또는 환경변수
	□ .env 파일 Git 미포함
	□ DB 포트 외부 미노출 (Security Group)
	□ Spring Actuator /env, /beans 외부 미노출
	□ H2 Console 운영 비활성화
	
네트워크:
	□ HTTPS 강제 (HTTP → HTTPS 리다이렉트)
	□ 불필요한 포트 Security Gorup에서 닫힘
	□ DB/Redis는 Private Subnet에만 위치
```
