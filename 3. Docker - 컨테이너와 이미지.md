
### 1-1. 로컬 환경 문제

```
개발자 로컬: Java 17, MySQL 8.0, Redis 7.0
스테이징 서버: Java 11, MySQL 5.7, Redis 6.2
운영 서버: Java 17, MySQL 8.0, Redis 7.0

→ 환경 차이로 로컬에서 되는 코드가 스테이징에서 안 됨
→ 환경 셋업 문서가 항상 최신이 아님
→ 팀원이 새로 합류하면 환경 셋업에 하루가 걸림
```

### 1-2. Docker 해결 방식

```
애플리케이션 + 실행 환경을 하나의 이미지로 패키징

이미지 안에 포함:
	- 애플리케이션 코드 (jar, 바이너리)
	- 런타임 (JDK 17)
	- 의존 라이브러리
	- 설정 파일
	  
이미지 하나를 어디서든 동일하게 실행 → 환경 차이 없음
```

## 2. 컨테이너 vs VM

```
가상머신 (VM):
  ┌──────────────────────────────────────┐
  │         Host OS (Linux)              │
  │  ┌────────────────────────────────┐  │
  │  │        Hypervisor              │  │
  │  │  ┌──────────┐  ┌──────────┐  │  │
  │  │  │  Guest OS│  │  Guest OS│  │  │
  │  │  │  (Linux) │  │(Windows) │  │  │
  │  │  │   App A  │  │   App B  │  │  │
  │  │  └──────────┘  └──────────┘  │  │
  │  └────────────────────────────────┘  │
  └──────────────────────────────────────┘
  - Guest OS 각각 수백 MB ~ 수 GB
  - 부팅 시간: 수십 초 ~ 수 분

컨테이너 (Docker):
  ┌──────────────────────────────────────┐
  │         Host OS (Linux)              │
  │  ┌─────────────────────────────────┐ │
  │  │       Docker Engine             │ │
  │  │  ┌──────────┐  ┌──────────┐   │ │
  │  │  │Container │  │Container │   │ │
  │  │  │  App A   │  │  App B   │   │ │
  │  │  └──────────┘  └──────────┘   │ │
  │  └─────────────────────────────────┘ │
  └──────────────────────────────────────┘
  - OS 커널 공유 → 수 MB 수준
  - 시작 시간: 수백 ms

컨테이너는 프로세스 격리 (namespace, cgroup)로 구현됨
```

## 3. 핵심 개념

```
이미지 (Image):
	컨테이너를 만들기 위한 읽기 전용 템플릿 (설계또)
	레이어(Layer) 구조로 저장됨 - 변경된 레이어만 재빌드
	
컨테이너 (Container):
	이미지를 실행할 인스턴스 (프로세스)
	여러 컨테이너를 같은 이미지로 실행 가능
	
레지스트리 (Registry):
	이미지 저장소
	- Docker Hub: 공개 레지스트리
	- AWS ECR, GCP Artifact Registry: 프라이빗 레지스트리
	  
이미지 태그:
	nginx:latest            → 최신 버전 (권장하지 않음, 버전 고정이 안전)
	nginx:1.25.3            → 특정 버전
	nginx:1.25.3-alpine     → Alpine Linux 기반 (경량)
```

## 4. Dockerfile

### 4-1. 기본 문법

```dockerfile
# 베이스 이미지 (FROM)
FROM eclipse-temurin:17-jre-alpine

# 메타데이터 (LABEL)
LABEL maintainer="hong@example.com"

# 환경변수 (ENV)
ENV APP_HOME=/app \
	SPRING_PROFILES_ACTIVE=prod
	
# 작업 디렉토리 (WORKDIR) - 이후 명령의 기준 경로
WORKDIR ${APP_HOME}

# 파일 복사 (COPY)
COPY build/libs/app.jar app.jar

# 포트 노출 선언 (EXPOSE) - 실제 포트 바인딩은 docker run에서
EXPOSE 8080

# 실행 시 명령 (ENTRYPOINT vs CMD)
ENTRYPOINT ["java", "-jar"]
CMD ["app.jar"]
# → docker run image other.jar 하면 CMD만 교체됨
```

### 4-2. Spring Boot Dockerfile - 레이어 최적화

```dockerfile
# 1단계: 빌드 (Multi-Stage Build)
FROM gradle:8-jdk17-alpine AS builder
WORKDIR /workspace
COPY build.gradle settings.gradle ./
COPY gradle ./gradle
# 의존성 먼저 받아두기 (캐시 활용)
RUN gradle dependencies --no-daemon
COPY src ./src
RUN gradle bootJar --no-daemon

# 2단계: 실행 이미지 (빌드 도구 제외 → 이미지 크기 최소화)
FROM eclipse-termurin:17-jre-alpine
WORKDIR /app

# 보안: root가 아닌 일반 유저로 실행
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# Spring Boot Layered Jar (의존성 레이어 분리)
COPY --from=builder /workspace/build/libs/app.jar app.jar
RUN java -Djarmode=layertools -jar app.jar extract

# 레이어 순서: 변경 빈도 낮은 것 먼저 (캐시 효율)
COPY --from=builder /workspace/build/libs/app.jar ./
EXPOSE 8080

# JVM 튜닝 (컨테이너 메모리 인식)
ENTRYPOINT ["java", \
	"-XX:+UseContainerSupport", \
	"-XX:MaxRAMPercentage=75.0", \
	"-jar", "app.jar"]
```

## 5. 주요 Docker 명령어

### 5-1. 이미지 관련

```bash
# 이미지 빌드
docker build -t myapp:1.0.0 .
docker build -t myapp:1.0.0 -f docker/Dockerfile .   # Dockerfile 경로 지정

# 이미지 목록
docker images

# 이미지 삭제
docker rmi myapp:1.0.0

# 레지스트리에 push/pull
docker tag myapp:1.0.0 123456789.dkr.ecr.ap-northeast-2.amazonaws.com/myapp:1.0.0
docker push 123456789.dkr.ecr.ap-northeast-2.amazonaws.com/myapp:1.0.0
docker pull nginx:1.25.3
```

### 5-2. 컨테이너 실행

```bash
# 기본 실행
docker run nginx:1.25.3

# 포트 바인딩: 호스트:컨테이너
docker run -p 8080:8080 myapp:1.0.0

# 백그라운드 실행 + 이름 지정
docker run -d --name myapp -p 8080:8080 myapp:1.0.0

# 환경변수 주입
docker run -d \
	-e SPRING_PROFILES_ACTIVE=prod \
	-e DB_URL=jdbc:postgresql://db.5432/mydb \
	-e DB_PASSWORD=secret \
	myapp:1.0.0
	
# 볼륨 마운트 (데이터 영속성)
docker run -d \
	-v /host/data:/container/data \
	-v myvolume:/var/lib/mysql \
	mysql:8.0
	
# 리소스 제한
docker run -d \
	--memory="512m" \
	--cpus="1.0" \
	myapp:1.0.0
```

### 5-3. 컨테이너 관리

```bash
# 실행 중인 컨테이너 목록
docker ps
docker ps -a  # 종료된 것까지 포함

# 로그 확인
docker logs myapp
docker logs -f myapp         # 실시간 스트리밍
docker logs --tail 100 myapp # 최근 100줄

# 컨테이너 내부 접속
docker exec -it myapp sh     # alpine
docker exec -it myapp bash   # ubuntu/debian

# 컨테이너 상태 (CPU, 메모리)
docker stats myapp

# 중지 / 삭제
docker stop myapp
docker rm myapp
docker stop myapp && docker rm myapp # 한 번에
```

## 6. 이미지 레이어 구조

```
FROM eclipse-temurin:17-jre-alpine  → Layer 1 (JRE 베이스)
RUN adduser appuser                 → Layer 2 (유저 생성)
COPY app.jar .                      → Layer 3 (애플리케이션)

이미지 = Layer 1 + Layer 2 + Layer 3 의 스택

레이어 캐시:
	- 한 번 빌드된 레이어는 캐시됨
	- 레이어가 바뀌면 그 이후 레이어를 모두 재빌드
	- COPY app.jar는 가장 자주 바뀌므로 맨 마지막에
	  
나쁜 예 (캐시 최적화 안 됨):
	COPY . .          # 소스 전체 복사 (자주 바뀜)
	RUN gradle build  # 의존성 다운로드 (시간 오래 걸림)
	→ 코드 한 줄 바꿔도 의존성 다운로드부터 재실행
	
좋은 예 (캐시 최적화):
	COPY build.gradle .     # 의존성 파일만 먼저 (변경 적음)
	RUN gradle dependencies 
	COPY src .              # 소스 나중에 (변경 잦음)
	RUN gradle build
	→ 의존성 변경 없으면 dependencies 단계 캐시 사용
```

## 7. 볼륨 (Volume)

```
컨테이너는 종료 시 네부 데이터가 사라짐
데이터를 영속적으로 보관하려면 볼륨 사용

종류:
	Volume:      Docker가 관리하는 볼륨 (권장)
	Bind Mount:  호스트 디렉토리를 컨테이너에 마운트
	tmpfs Mount: 메모리에 임시 마운트
	
# Docker Volume
docker volume create mydata
docker run -v mydata:/var/lib/mysql mysql:8.0

# Bind Mount (개발 시 유용: 코드 변경 즉시 반영)
docker run -v $(pwd)/src:/app/src myapp:dev

# Volume 목록 / 삭제
docker volume ls
docker volume rm mydata
docker volume prune  # 사용 안 하는 볼륨 정리
```

## 8. 자주 쓰는 패턴

```bash
# 실행 중인 모든 컨테이너 중지
docker stop $(docker ps -q)

# 종료된 컨테이너 전체 삭제
docker container prune

# 이미지/컨테이너/볼륨 전체 정리 (주의!)
docker system prune -a

# 컨테이너 내 특정 파일 복사
docker cp myapp:/app/logs/error.log ./error.log

# 이미지 레이어 분석 (크기 확인)
docker history myapp:1.0.0

# 컨테이너 IP 확인
docker inspect myapp | grep IPAddress
```
