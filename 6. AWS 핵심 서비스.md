```
인터넷
  │
  ▼
Route 53 (DNS)
  │
  ▼
CloudFront (CDN) ────→ S3 (정적 파일)
  │
  ▼
ALB (Application Load Balancer)
  │
  ├──→ ECS/EKS (컨테이너 실행)
  │      │
  │      ├──→ RDS (MySQL/PostgreSQL)
  │      ├──→ ElastiCache (Redis)
  │      └──→ MSK (Kafka)
  │
  └──→ Lambda (서버리스 함수)

공통:
  ECR (컨테이너 이미지 저장소)
  Secrets Manager (비밀값 관리)
  CloudWatch (로그, 메트릭, 알람)
  IAM (권한 관리)
```

## 2. IAM - 권한의 시작

### 2-1. 개념

```
IAM (Identity and Access Management):
	AWS 리소스에 대한 접근을 제어

핵심 개념:
	User:   사람이 로그인하는 계정
	Group:  User의 묶음 (개발팀, 운영팀)
	Role:   AWS 서비스가 다른 서비스에 접근할 때 (EC2 → S3 등)
	Policy: JSON으로 표현된 권한 정의
	
원칙: 최소 권한 원칙 (Least Privilege)
	 필요한 권한만, 필요한 시간에만 부여
```

### 2-2. Policy 예시

```json
{
	"Version": "2012-10-17",   // 정책 문법(스키마) 버전 - 보통 이 날짜 사용함
	"Statement": [             // 규칙 배열
		{
			"Effect": "Allow",        // 허용 규칙 (Allow or Deny)
			"Action": [
				"s3:GetObject",
				"s3:PutObject"
			],
			"Resource": "arn:aws:s3:::my-bucket/*"   // Action이 적용될 대상 리소스
		},
		{
			"Effect": "Deny",         // 거부 규칙 (Allow or Deny)
			"Action": "s3:DeleteObject", // S3 객체 삭제 권한을 명시적으로 차단
			"Resource": "arn:aws:s3:::my-bucket/*"
		}
	]
}
```

### 2-3. EC2 / ECS 에서 IAM Role 사용

```
ECS Task가 S3에 접근해야 할 때:
	Access Key를 코드에 박지 않음 → 보안 취약
	대신: ECS Task Role을 생성하고 S3 접근 권한 부여
	→ 컨테이너 내에서 자동으로 임시 자격증명 획득
	
Spring Boot에서:
	AWS SDK가 자동으로 환경에서 자격증명 탐색
	(코드에 키를 하드코딩할 필요 없음)
```

## 3. VPC - 네트워크 격리

### 3-1. 구조

```
VPC (10.0.0.0/16)
  │
  ├── Public Subnet (10.0.1.0/24) — AZ-a
  │   └── ALB, NAT Gateway, Bastion Host
  │
  ├── Public Subnet (10.0.2.0/24) — AZ-b
  │
  ├── Private Subnet (10.0.11.0/24) — AZ-a
  │   └── ECS/EKS Worker Node, EC2
  │
  ├── Private Subnet (10.0.12.0/24) — AZ-b
  │
  ├── Database Subnet (10.0.21.0/24) — AZ-a
  │   └── RDS, ElastiCache
  │
  └── Database Subnet (10.0.22.0/24) — AZ-b
  
인터넷 게이트웨이: Public Subnet <> 인터넷
NAT 게이트웨이: Private Subnet → 인터넷 (단방향, 업데이트용)
```

### 3-2. Security Group - 방화벽

```
ALB Security Group:
	Inbound: 0.0.0.0/0 → 80, 443 허용 (인터넷 트래픽)
	Outbound: ECS SG → 8080 허용
	
ECS Security Group:
	Inbound: ALB SG → 8080 허용 (ALB에서만)
	Outbound: RDS SG → 3306, Redis SG → 6379 허용
	
RDS Security Group:
	Inbound: ECS SG → 3306 허용 (ECS에서만)
	Outbound: 없음
```

## 5. ECS - 컨테이너 실행 서비스

### 5-1. ECS vs EKS

```
ECS (Elastic Container Service):
	- AWS 네이티브 컨테이너 오케스트레이션
	- 설정이 단순 (쿠버네티스보다 러닝 커브 낮음)
	- AWS 서비스와 깊게 통합 (IAM, CloudWatch, ALB)
	- Fargate와 함께 사용 시 서버 관리 불필요
	  
EKS (Elastic Kubernetes Service):
	- 관리형 쿠버네티스
	- 쿠버네티스 생태계 그대로 사용 (Helm, Istio 등)
	- 이식성 (다른 클라우드/온프레미스로 이전 가능)
	- 더 복잡한 설정, 더 많은 기능
```

## 6. RDS - 관리형 DB

```
RDS가 처리해주는 것:
	- 자동 백업 (일별 스냅샷, 트랜잭션 로그)
	- 패치 / 업그레이드 (유지보수 윈도우 설정)
	- 다중 AZ (Multi-AZ): 자동 페일오버
	- Read Replica: 읽기 스케일 아웃
	- 모니터링: CloudWatch 연동
	  
Multi-AZ 구성:
	Primary (AZ-a) --동기복제--> Standby (AZ-b)
	Primary 장애 시 → 자동으로 Standby가 Primary 승격
	RPO ~~ 0, RTO ~~ 1~2분
	
Read Replica:
	Primary --비동기 복제--> Replica 1 (읽기 전용)
	                      → Replica 2 (읽기 전용)
	쓰기 → Primary, 읽기 → Replica
	복제 지연(Replication Lag) 주의
```

```yaml
# Spring Boot RDS 연결 (Secrets Manager 사용)
spring:
	datasource:
		url: jdbc:mysql://${DB_HOST}:3306/${DB_NAME}
		username: ${DB_USERNMAE}
		password: ${DB_PASSWORD}
		hikari:
			maximum-pool-size: 10
			minimum-idle: 5
			connection-timeout: 30000
			idle-timeout: 60000
			max-lifetime: 1800000
```

## 7. ElastiCache - 관리형 Redis

```
ElastiCache Redis 운영 설정:

클러스터 모드 비활성화 (단일 샤드):
	- Primary + Replica 구성
	- 페일오버 자동화

클러스터 모드 활성화:
	- 여러 샤드로 데이터 분산
	- 대용량 처리에 적합
	  
주의:
	ElastiCache는 VPC 내부에서만 접근 가능
	로컬 개발 시: SSH 터널링 또는 Bastion Host 활용
```

```bash
# SSH 터널로 로컬에서 ElastiCache 접근 (개발 시)
ssh -L 6379:my-redis.xxxxx.cache.amazonaws.com:6379 ec2-user@bastion-host -N
# → localhost:6379로 접근하면 ElastiCache로 터널링
```

## 8. S3 - 오브젝트 스토리지

```bash
# 버킷 생성 및 정책 설정
aws s3 mb s3://my-app-uploads --region ap-northeast-2

# 파일 업로드/다운로드
aws s3 cp ./file.pdf s3://my-app-uploads/documents/file.pdf
aws s3 cync ./local-dir s3://my-app-uploads/backups/

# 버킷 정책 - CloudFront만 접근 허용
{
	"Version": "2012-10-17",
	"Statement": [{
		"Effect": "Allow",
		"Principal": {
			"Service": "cloudfront.amazonaws.com"
		},
		"Action": "s3:GetObject",
		"Resource": "arn:aws:s3:::my-app-uploads/*",
		"Condition": {
			"StringEquals": {
				"AWS:SourceArn":"arn:aws:cloudfront::123:distribution/XXXXX"
			}
		}
	}]
}
```

```java
// Spring Boot에서 S3 Presigned URL 생성 (직접 업로드)
@Service
public class S3Service {
	private final S3Presigner presigner;
	
	// 클라이언트가 직접 S3에 업로드할 수 있는 서명된 URL 생성
	public String generatePresignedUploadUrl(String key) {
		PutObjectPresignRequest request = PutObjectPresignRequest.builder()
			.signatureDuration(Duration.ofMinutes(10))	
			.putObjectRequest(r -> r
				.bucket("my-app-uploads")
				.key(key)
				.contentType("image/jpeg"))
			.build();
			
		return presigner.presignPutObject(request).url().toString();
	}
}
```

## 9. CloudWatch — 로그와 모니터링

```bash
# 로그 그룹 생성
aws logs create-log-group --log-group-name /app/myapp --region ap-northeast-2

# 로그 보존 기간 설정 (30일)
aws logs put-retention-policy \
  --log-group-name /app/myapp \
  --retention-in-days 30

# 로그 스트림 실시간 조회
aws logs tail /app/myapp --follow

# 특정 시간대 로그 조회
aws logs filter-log-events \
  --log-group-name /app/myapp \
  --start-time $(date -d '1 hour ago' +%s)000 \
  --filter-pattern "ERROR"
```

## 10. Secrets Manager — 비밀값 관리

```bash
# 비밀값 저장
aws secretsmanager create-secret \
  --name myapp/prod/db \
  --secret-string '{"username":"appuser","password":"supersecret","host":"my-db.xxxxx.rds.amazonaws.com"}'

# 조회
aws secretsmanager get-secret-value --secret-id myapp/prod/db

# 자동 교체 (30일마다 비밀번호 변경)
aws secretsmanager rotate-secret \
  --secret-id myapp/prod/db \
  --rotation-lambda-arn arn:aws:lambda:... \
  --rotation-rules AutomaticallyAfterDays=30
```

```java
// Spring Boot에서 Secrets Manager 자동 로드
// spring-cloud-aws-secrets-manager-config 의존성 추가 후
// application.yml
spring:
  config:
    import: "aws-secretsmanager:myapp/prod/db"
  datasource:
    url: jdbc:mysql://${host}:3306/mydb
    username: ${username}
    password: ${password}   # Secrets Manager에서 자동 주입
```

## 11. AWS 비용 절감 팁

```
EC2/ECS:
	Savings Plans / Reserved Instance: 1~3년 약정 시 최대 72% 절감
	Spot Instance: 최대 90% 절감 (언제든 회수될 수 있음 → Stateless 워크로드에)
	
RDS:
	Reserved Instance: 1~3년 약정 시 최대 60% 절감
	Aurora Serverless v2: 사용한 만큼만 과금 (개발/테스트 적합)
	
비용 모니터링:
	AWS Cost Explorer: 서비스별 비용 분석
	AWS Budgets: 예산 초과 시 알림
	Cost Anomaly Detection: 비정상적 비용 급증 감지
	
태그 전략:
	모든 리소스에 Environment, Team, Service 태그 부착
	→ 비용을 팀/서비스별로 추적 가능
```
